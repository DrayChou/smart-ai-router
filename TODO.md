## 🔍 发现的架构问题和改进方向

### 🚨 P0 - 渠道级别定价架构问题 (Critical Architecture Issue)
**问题描述**: 不同API Key对应不同用户级别，但系统按渠道缓存定价信息，导致定价错误

**问题现象**:
- 同一渠道的免费用户 vs 付费用户有不同定价和可用模型
- SiliconFlow、OpenRouter、Gemini等都存在此问题
- 当前按 `channel_id` 缓存，没有区分 `api_key` 级别

**风险评估**:
- **🔴 高风险**: 付费用户可能按免费价格计费
- **🔴 高风险**: 免费用户可能尝试访问付费专属模型  
- **🔴 高风险**: 价格计算完全错误，影响成本控制准确性

**解决方案设计**:
```python
# 当前架构 (问题)
model_cache[channel_id] -> 单一的模型列表和定价

# 目标架构 (解决方案)  
model_cache[f"{channel_id}_{api_key_hash}"] -> 每个API Key独立缓存
```

**实施计划**:
- [ ] **Phase 8.1**: 设计API Key级别的缓存架构
- [ ] **Phase 8.2**: 修改模型发现任务，支持多Key发现
- [ ] **Phase 8.3**: 更新路由逻辑，使用Key对应的缓存
- [ ] **Phase 8.4**: 实现缓存清理和Key管理机制

**影响评估**:
- **兼容性**: 单Key渠道保持现有行为
- **性能**: 存储增加，需要优化
- **复杂度**: 需要Key级别的状态管理
